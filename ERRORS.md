## Возможные ошибки и их решение

### 1. Ошибка компиляции: `--use-port=zstd` не работает
**Причина:** В некоторых версиях Emscripten порт `zstd` отсутствует.  
**Решение:** Используйте локальную компиляцию с исходниками Zstd, как показано в `build.sh`.

### 2. `Module.HEAPU8` не определён после загрузки WASM
**Причина:** Не экспортированы runtime-методы.  
**Решение:** Добавьте при компиляции флаг `-s EXPORTED_RUNTIME_METHODS='["HEAPU8","HEAP8","wasmMemory"]'`.

### 3. Ошибка `detached buffer` при вызове `set`
**Причина:** Кеширование ссылки на `HEAPU8` до расширения памяти.  
**Решение:** Всегда используйте актуальный `Module.HEAPU8` непосредственно перед операциями, не сохраняйте его в переменную.

### 4. Сконвертированный NSP не запускается
**Причина:** Отсутствие выравнивания (padding) данных по границе 0x200 байт, требуемого форматом PFS0.  
**Решение:** В код добавлено вычисление и запись padding’а перед каждым файлом и после него. Убедитесь, что используется последняя версия `nsz2nsp.cpp` с поддержкой выравнивания.

### 5. Ошибка CORS при открытии `file://`
**Причина:** Браузер блокирует загрузку WASM из локальных файлов.  
**Решение:** Запустите локальный HTTP-сервер и открывайте страницу через `http://localhost`.

### 6. Ошибка `Invalid NSZ file`
**Причина:** Файл не является корректным NSZ (отсутствует сигнатура `NSZ0`) или повреждён.  
**Решение:** Проверьте файл с помощью оригинальной утилиты `nsz`. Если файл в порядке, возможно, он имеет нестандартную структуру (например, другую версию формата).

### 7. Не хватает памяти для больших файлов (3+ ГБ)
**Причина:** Ограничение памяти браузера.  
**Решение:** Используйте современный браузер с большим лимитом памяти (например, Chrome с флагом `--max-old-space-size=8192`). Флаг `ALLOW_MEMORY_GROWTH=1` позволяет модулю расширять память по мере необходимости.

### 8. Ошибка `TypeError: Module._malloc is not a function`
**Причина:** Функции не экспортированы или модуль не загрузился.  
**Решение:** Проверьте, что файл `nsz2nsp.wasm` загружается (вкладка Network в инструментах разработчика) и что в `EXPORTED_FUNCTIONS` указаны правильные имена (с подчёркиванием).

### 9. Ошибка `memory access out of bounds` при конвертации NSZ с заголовком PFS0, но нулевой строковой таблицей
**Причина:** Файл имеет нестандартную структуру (старая версия NSZ или повреждённый заголовок), где поле `string_table_size` в заголовке PFS0 равно нулю, хотя количество файлов (`file_count`) больше нуля. Это приводит к выходу за границы памяти при попытке чтения имён файлов.  
**Решение:** В код добавлена эвристическая логика, аналогичная той, что используется в оригинальной утилите `nsz` начиная с версии 4.4.0. Если размер строковой таблицы указан как 0, программа пытается определить её реальный размер, вычисляя разницу между смещением первого файла и концом области записей. Это позволяет корректно обрабатывать такие "битые" файлы.

### 10. Проблемы с бит-идентичным восстановлением NSP из NSZ
**Причина:** В старых версиях NSZ и некоторых других инструментах мог использоваться нестандартный паддинг (пустое пространство) в таблице записей файлов (FileEntryTable), что приводило к невозможности точного восстановления исходного NSP.  
**Решение:** Начиная с версии 4.5.0, оригинальный NSZ сохраняет точный размер таблицы записей, включая любой нестандартный паддинг. В нашем WebAssembly-конвертере добавлена аналогичная логика: по умолчанию сохраняется оригинальный размер таблицы записей.

### 11. Ошибка "String table missing null terminator" при конвертации старых NSZ
**Причина:** Многие старые инструменты создания NSZ не добавляли обязательный нулевой терминатор в конец строковой таблицы, что нарушает спецификацию PFS0 и вызывает ошибки при чтении имён файлов. Эта проблема была зафиксирована в **issue #151** и исправлена в NSZ 4.6.0.  
**Решение:** В наш конвертер добавлена автоматическая проверка наличия нулевого терминатора. Если он отсутствует, программа выдаёт предупреждение и при записи выходного NSP автоматически добавляет недостающий терминатор, корректируя все смещения файлов.

### 12. Ошибка "PFS0 header overlaps with first file"
**Причина:** При использовании опции `--fix-padding` (или в некоторых старых файлах) заголовок PFS0 мог перекрываться с первым файлом из-за неверного выравнивания.  
**Решение:** В коде добавлена проверка перекрытия заголовка с первым файлом. Если оно обнаружено, смещение первого файла корректируется добавлением необходимого паддинга (0x200 байт).

### 13. Обработка файлов с мусорными/невалидными записями
**Причина:** Некоторые NSZ-файлы (особенно старые или созданные некорректными инструментами) содержат в таблице записей PFS0 «мусор» — записи с нулевым размером, невалидными смещениями или отсутствующей сигнатурой NCZ для `.ncz`-файлов. Попытка обработать такие записи приводит к выходу за границы памяти.  
**Решение:** В код добавлена фильтрация записей при чтении. Каждая запись проверяется на:
- Выход за границы входного файла (`data_offset + data_size <= input_size`)
- Ненулевой размер
- Наличие сигнатуры `NCZ` для файлов с расширением `.ncz` (с учётом спецификации формата, где сигнатура находится после заголовка NCA размером 0x4000 байт)
Только записи, прошедшие все проверки, включаются в выходной NSP. Это позволяет корректно обрабатывать файлы, содержащие до 160 записей, из которых реальны лишь несколько (например, 4). Остальные игнорируются, и выходной NSP создаётся только из валидных данных.

### 14. NCZ-файл не распознаётся и копируется как есть (вместо распаковки)
**Причина:** В старых форматах NCZ сигнатура `NCZ` находится не в начале файла, а после заголовка NCA размером 0x4000 байт. Простая проверка первых байт не находит её, и файл ошибочно считается несжатым.  
**Решение:** В обновлённой версии конвертера реализована полная спецификация NCZ:
- Читается и сохраняется заголовок NCA (первые 0x4000 байт)
- Проверяется магическое число `0x4E544345535A434E` ("NCZSECTN") после заголовка NCA
- Пропускаются секции заголовка NCZ (количество секций × 0x30 байт)
- Оставшиеся данные декомпрессируются Zstandard
- Выходной NCA формируется склеиванием сохранённого заголовка и распакованных данных
Это позволяет корректно обрабатывать NCZ-файлы любого формата.

### 15. Несоответствие выходного размера (ошибка `output size mismatch`)
**Причина:** Для NCZ-файлов точный размер после распаковки неизвестен до фактической декомпрессии. Предварительный расчёт может дать недостаточный или избыточный размер.  
**Решение:** В конвертере используется двухпроходной подход:
- На этапе `nsz_get_output_size` используется приблизительный размер (например, сжатый размер × 2) или размер, полученный из заголовка (если доступен)
- На этапе `nsz_convert` после распаковки фактический размер сравнивается с зарезервированным; при нехватке памяти возвращается ошибка
- Для большинства файлов размер угадывается с запасом, и конвертация проходит успешно
